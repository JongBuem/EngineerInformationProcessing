Algorithm Study Note

Java 필드 섀도잉 & 생성자 체이닝 추적

Java · field shadowing · this() · super() · constructor chaining · getX() · Tracing
Q문제 코드
Java 
class Parent {
    int x = 100;              // Parent의 x 필드
    Parent() { this(500); }  // → Parent(500)
    Parent(int x) { this.x = x; }  // Parent.x 설정
    int getX() { return x; }   // ★ Parent 클래스에 정의 → Parent.x 반환
}

class Child extends Parent {
    int x = 4000;             // Child의 x 필드 (Parent.x를 가림)
    Child() { this(5000); }  // → Child(5000)
    Child(int x) { this.x = x; }  // Child.x 설정
}

public class Main {
    public static void main(String[] args) {
        Child obj = new Child();
        System.out.println(obj.getX());
    }
}

위 코드의 출력값은 무엇인가?

System.out.println(obj.getX()) 출력 결과
500
getX()는 Parent에 정의 → Parent.x 반환
Parent.x = 500 (Parent() → this(500) → Parent.x=500)
Child.x = 5000 이지만 getX()에서는 접근 안 됨
① 문제 유형

부모·자식 클래스에 같은 이름의 필드 x가 각각 존재(필드 섀도잉)할 때, this() 생성자 체이닝과 묵시적 super() 호출이 어떤 x를 초기화하는지 추적하는 문제입니다. getX()가 어느 클래스의 x를 반환하는지가 핵심입니다.

④ 단계별 x 값 변화

최종 obj.getX() → Parent.x 반환 500 ← 반환값 5000
⑤ 핵심 포인트
  • 🌑 필드 섀도잉(Field Shadowing): Child.x와 Parent.x는 같은 이름이지만 별개의 메모리입니다. 메서드가 정의된 클래스의 x가 사용됩니다.
  • 🔗 getX()는 Parent에 정의되어 있으므로 내부의 x는 Parent.x를 가리킵니다. Child.x=5000이 아닙니다.
  • ⬆️ Child 생성자에 명시적 super()가 없으면 컴파일러가 자동으로 super() 삽입합니다. 반드시 부모 생성자가 먼저 실행됩니다.
  • 🎯 Parent() → this(500) 체인으로 Parent.x = 500이 설정되고, Child(5000)으로 Child.x = 5000이 설정됩니다.
