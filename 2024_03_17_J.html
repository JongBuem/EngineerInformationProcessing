Java 다형성 — 메서드 동적 디스패치 vs 필드 섀도잉

Java · OOP · Polymorphism · @Override · Field Shadowing · Dynamic Dispatch · Tracing
Q문제 코드
Java 
class B {
    int x = 3;
    int getX() { return x * 2; }   // B의 x(3) * 2 = 6
}

class A extends B {
    int x = 7;                         // B의 x를 가림(field shadowing)
    @Override
    int getX() { return x * 3; }   // A의 x(7) * 3 = 21
}

public class Annotation {
    public static void main(String[] args) {
        B b1 = new A();  // 선언타입 B, 실제객체 A
        A b2 = new A();  // 선언타입 A, 실제객체 A
        System.out.print(b1.getX() + b1.x + b2.getX() + b2.x);
    }
}

위 코드의 출력값은 무엇인가?

System.out.print() 출력 결과
52
21 + 3 + 21 + 7 = 52
① 문제 유형

메서드는 실제 객체 타입으로 동적 디스패치되지만, 필드(인스턴스 변수)는 선언 타입 기준으로 접근되는 Java 다형성의 핵심 차이를 파악하는 문제입니다.


  • ⚠️핵심 함정: B b1 = new A()에서 b1.getX()는 A의 메서드(동적 디스패치)지만, b1.x는 선언 타입 B의 필드(x=3)입니다.
  • 🟢메서드 @Override: 런타임 실제 객체 타입 기준 → 항상 A의 getX() 실행
  • 🔴필드 섀도잉: 컴파일 타임 선언 타입 기준 → B b1이면 B.x(3), A b2이면 A.x(7)
② 메서드 vs 필드 접근 규칙
✅ 메서드 — 동적 디스패치 (런타임)
실제 객체(new A())의 타입 기준
선언 타입이 B여도
→ A의 getX() 호출됨

b1.getX() = A의 getX() = 21
⚠️ 필드 — 선언 타입 기준 (컴파일)
선언된 참조 타입 기준
B b1 → B.x = 3
A b2 → A.x = 7

필드는 오버라이딩 안 됨!

④ 각 항 단계별 계산 추적
항 선언 타입 실제 객체 접근 방식 사용 값 결과

⑤ 핵심 주의 포인트
  • 🟢 메서드는 런타임 실제 타입 기준 (동적 디스패치): B b1 = new A()여도 b1.getX()는 A의 오버라이딩된 메서드를 호출합니다.
  • 🔴 필드는 컴파일 타임 선언 타입 기준 (필드 섀도잉): B b1 = new A()에서 b1.x는 B의 x(=3)입니다. A의 x(=7)가 아닙니다!
  • 💡 b1과 b2는 모두 new A()로 같은 실제 객체지만, 선언 타입이 다를 때 필드 접근 결과가 달라집니다.
