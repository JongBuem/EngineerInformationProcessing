1this() — 생성자 체이닝

this()는 같은 클래스의 다른 생성자를 호출합니다. 생성자 중복 코드를 줄일 때 사용하며, 반드시 생성자의 첫 번째 줄에 위치해야 합니다.

Java 
class Box {
    int w, h, d;

    Box() {
        this(1, 1, 1);      // ① Box(int,int,int) 호출 — 첫 줄 필수
    }
    Box(int w, int h) {
        this(w, h, 1);     // ② 세 인자 버전으로 위임
    }
    Box(int w, int h, int d) {
        this.w = w;  this.h = h;  this.d = d;  // 실제 초기화
    }
}

// new Box()   → this(1,1,1) → Box(1,1,1): w=1,h=1,d=1
// new Box(2,3) → this(2,3,1) → Box(2,3,1): w=2,h=3,d=1
★실행 흐름 — 시험 출제 패턴
1
new Box() 호출
Box() 생성자 진입
↓ this(1,1,1)
2
Box(int,int,int) 진입
this.w=1, this.h=1, this.d=1 실행 ✓
↩ 반환 후 Box() 본문 없음 → 완료

규칙: this()와 super()는 동시에 사용 불가, 한 생성자에 하나만 가능하며 둘 다 첫 줄이어야 합니다.

2super() — 부모 생성자 자동/명시 호출

자식 생성자에 super()나 this()를 명시하지 않으면 컴파일러가 super()를 첫 줄에 자동 삽입합니다. 이 때 부모에 기본 생성자(매개변수 없는)가 없으면 컴파일 에러!

Java — super() 자동 삽입 
class Parent {
    int x;
    Parent()    { x = 10; }       // 기본 생성자
    Parent(int v) { x = v; }        // 매개변수 생성자
}

class Child extends Parent {
    int y;
    Child(int v) {
        // super(); ← 컴파일러 자동 삽입 → Parent.x = 10
        y = v;
    }
    Child(int px, int cy) {
        super(px);  // 명시 — Parent(int) → x = px
        y = cy;
    }
}

// new Child(5)     → super() → x=10, y=5
// new Child(3, 7)  → super(3) → x=3,  y=7
상황 동작
자식 생성자에 아무것도 없음 컴파일러 → super() 자동 삽입
this()만 있음 super()는 최종 생성자에서 자동 삽입
super(args) 명시 해당 부모 생성자 직접 호출
부모에 기본 생성자 없음 + 자동 삽입 시도 ❌ 컴파일 에러
3필드 섀도잉 (Field Shadowing) ★ 최빈출

부모와 자식에 같은 이름의 필드가 있으면, 자식의 필드가 부모의 필드를 "가립니다(shadow)". 하지만 메모리는 각각 별개로 존재합니다. 메서드가 정의된 클래스의 필드가 사용됩니다.

Java — 필드 섀도잉 예시 
class Parent {
    int x = 100;
    int getX() { return x; }  // Parent.x 반환
}

class Child extends Parent {
    int x = 200;              // Parent.x를 가리는 별개 필드
    // getX() 오버라이딩 없음
}

Child c = new Child();
System.out.println(c.x);          // 200  ← Child.x
System.out.println(c.getX());     // 100  ← Parent.getX() 안의 x = Parent.x
System.out.println(((Parent)c).x); // 100  ← 참조 타입 캐스팅 시 Parent.x
★메모리 구조 — 두 x는 별개
Parent 영역
x=100
getX() → return this.x
= 100
Child 영역
x=200
c.x → Child.x
= 200
핵심 규칙: 필드는 참조 타입 기준 (컴파일 타임 결정)
메서드는 실제 타입 기준 (런타임 결정 — 동적 디스패치)
4동적 디스패치 — 참조 타입 vs 실제 타입

Parent obj = new Child()처럼 부모 타입으로 자식 인스턴스를 참조할 때, 오버라이딩된 메서드는 실제 타입(Child) 버전이 호출됩니다. 하지만 필드와 static 메서드는 참조 타입 기준입니다.

Java — 동적 디스패치 완전 정리 
class Animal {
    String type = "Animal";
    void sound()   { System.out.print("...");  }
    static void tag() { System.out.print("A-tag"); }
}

class Dog extends Animal {
    String type = "Dog";              // 필드 섀도잉
    @Override
    void sound()   { System.out.print("Woof"); }  // 오버라이딩
    static void tag() { System.out.print("D-tag"); } // 숨김(hiding)
}

Animal obj = new Dog();
obj.sound();         // "Woof"   ← 런타임: Dog 버전 (동적 디스패치)
System.out.println(obj.type); // "Animal" ← 컴파일: 참조 타입 기준 (섀도잉)
Animal.tag();        // "A-tag"  ← static은 참조 타입 기준
구분 결정 시점 기준 예시 결과
오버라이딩 메서드 런타임 실제 타입 (Dog) Dog.sound() 실행
필드 (섀도잉) 컴파일 참조 타입 (Animal) Animal.type = "Animal"
static 메서드 (숨김) 컴파일 참조 타입 (Animal) Animal.tag() 실행
오버로딩 컴파일 참조 타입 (인자 타입) 선언 타입 기준
5자주 나오는 함정 포인트
① this() 체이닝 후 Parent.x vs Child.x

Child 생성자에서 this.x = val은 Child.x를 설정합니다. 상속받은 Parent.x는 super() 체인을 통해서만 설정됩니다. getX()가 어느 클래스에 정의되어 있냐에 따라 반환값이 달라집니다.

② 인자 없는 메서드 호출 — 어떤 버전?

obj.getName()처럼 인자 없는 호출은 자식에 해당 버전(매개변수 없는)이 없으면 부모의 구현이 실행됩니다. 오버로딩과 오버라이딩을 혼동하지 마세요.

③ abstract 클래스 — 인스턴스화 불가

new Vehicle()처럼 abstract 클래스를 직접 생성하면 컴파일 에러입니다. 반드시 자식 클래스를 통해 인스턴스화해야 합니다.

  • ⭐필드: 참조 타입 기준 (컴파일) / 오버라이딩 메서드: 실제 타입 기준 (런타임)
  • ⭐getX()가 Parent에 있으면 → Parent.x / Child에 오버라이딩 시 → Child.x
  • ⭐생성자 체인 순서: super()가 가장 먼저, 부모 → 자식 순으로 초기화
  • ⭐this() 체이닝은 같은 클래스 안에서만. super()는 부모 클래스로.
