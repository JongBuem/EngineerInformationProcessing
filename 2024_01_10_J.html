Java 상속 · super() · 메서드 오버로딩 실행 순서 추적

Java · Inheritance · super() · Overloading · Field Shadowing · Execution Order
Q문제 코드
Java 
class Parent {
    int x, y;

    Parent(int x, int y) { ①
        this.x = x;
        this.y = y;
    }

    int getT() { ②
        return x * y;
    }
}

class Child extends Parent {
    int x;                        // Parent의 x를 가림(field shadowing)

    Child(int x) { ③
        super(x + 1, x);          // super(4, 3)
        this.x = x;               // Child.x = 3
    }

    int getT(int n) { ④           // getT()의 오버로딩 (파라미터 다름)
        return super.getT() + n;
    }
}

class Main {
    public static void main(String[] args) { ⑤
        Parent parent = new Child(3);  ⑥
        System.out.println(parent.getT());  ⑦
    }
}

① 문제 유형

상속 관계에서 super() 생성자 체인, 필드 섀도잉(Field Shadowing), 메서드 오버로딩과 오버라이딩의 차이를 동시에 묻는 복합 추적 문제입니다.


  • ⚠️함정 1 — 오버로딩 vs 오버라이딩: Child의 getT(int n)은 파라미터가 달라 오버로딩입니다. Parent의 getT()는 오버라이드되지 않았습니다.
  • ⚠️함정 2 — 참조 타입이 Parent: Parent parent = new Child(3)이므로 parent.getT()는 인수 없는 버전 → Parent.getT() 호출.
  • ⚠️함정 3 — 필드 섀도잉: Parent.getT()가 접근하는 x는 Parent 레이어의 x=4입니다. Child.x=3은 별개 필드.

⑤ 오버로딩 vs 오버라이딩 — 핵심 구분
구분 Parent Child 관계
시그니처 getT() getT(int n) 오버로딩 (Overloading)
parent.getT() 호출 시 참조 타입 Parent의 getT() → Parent.getT() 실행 Parent.②
만약 오버라이딩이었다면 실제 타입 Child의 getT() → Child.getT() 실행 (동적 디스패치) 이 문제에서는 해당 없음
★ 오버로딩 판별 기준: 메서드 이름은 같지만 파라미터 개수/타입이 다르면 오버로딩.
getT() ≠ getT(int n) → Child는 Parent의 getT()를 오버라이드하지 않음.
따라서 parent.getT()는 동적 디스패치 없이 Parent.getT()가 호출됩니다.
⑥ 핵심 주의 포인트
  • 🔗 super() 체인: Child 생성자는 반드시 가장 먼저 super()를 호출합니다. Parent 생성자가 먼저 완료된 후 Child 나머지 코드가 실행됩니다.
  • 🎭 필드 섀도잉: Child.x=3과 Parent.x=4는 이름이 같지만 별개의 필드입니다. Parent.getT()는 Parent 레이어의 x(=4)를 사용합니다.
  • 🔢 Parent.getT() = x * y = 4 * 3 = 12
  • 📋 실행 순서: ⑤ → ⑥ → ③ → ① → ⑦ → ②
